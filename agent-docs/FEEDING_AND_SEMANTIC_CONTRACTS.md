# Feeding Mechanism and Semantic Contracts Integration

**Status:** Implementation Complete | Testing Phase  
**Audience:** Addon developers, API maintainers, MCP server operators  
**Last Updated:** April 3, 2026

---

## Overview

The strengthened feeding mechanism provides a **governance-backed intake system** for addon discoveries while maintaining strict separation from semantic contracts. This document explains:

1. How observations fit into contract-native architecture
2. Why observations are never the source of truth
3. Governance workflow (review → promote → regenerate)
4. Best practices for addon teams
5. Troubleshooting and validation patterns

---

## Architecture: Observations vs. Semantic Contracts

### Semantic Sources of Truth (Immutable)

These remain the **only authoritative semantic sources**:

| Source | Location | Content | Generated By |
|--------|----------|---------|--------------|
| **xml-tree** | `tools/api_doc_gen/` (XML parser logic) | XML element hierarchy, attributes, handlers | Parser phase 1: lexical+structural |
| **lua-analysis** | `tools/api_doc_gen/` (Lua analyzer logic) | Lua function signatures, scoping, control flow | Parser phase 2: semantic analysis |
| **xml-lua-links** | `tools/api_doc_gen/` (classifier logic) | Cross-validation between XML and Lua | Generator phase 3: correlation |

**Output:** `docs/war-api/` (canonical generated docs)

### Feeding System (Governed Input)

Acts as **structured feedback intake** for addon-discovered patterns:

| Phase | Content | Status | Audience | Can Affect Docs? |
|-------|---------|--------|----------|-----------------|
| **Ingest** | Raw `.feed.json` observation + evidence narrative `.md` | `candidate` | Dev who discovered pattern | No (not yet visible) |
| **Review** | Accept/reject verdict + reviewer notes | `reviewed` OR `rejected` | Maintainers/API experts | No (rejected is archived) |
| **Promote** | Observation moved to seed file with metadata | `promoted` | Seed maintainers | Yes (feeds tool/regen) |
| **Regenerate** | Seed content integrated into canonical docs | Final | Everyone | Yes (becomes canonical) |

**Output:** Seeds update `docs/platform/seeds/`, then docs regenerate

### Data Flow with Clear Separation

```
ADDON CODE
    ↓
OBSERVATION (Addon developer's claim with evidence)
    │
    ├─ Evidence: code_change, runtime_observation, user_report, test
    ├─ Claims: behavior_caveat, usage_pattern, signature_hint
    ├─ Symbols: informational references only
    └─ Confidence: explicit tracking (HIGH/MEDIUM/LOW)
    ├────→ Ingest (validate schema, store in queue)
    │        ↓
    │    Status: candidate
    │    Visibility: None (not in MCP queries)
    │    Impact: Zero on semantics
    │
    ├────→ Review (accept/reject decision)
    │        ├─ Accept → Status: reviewed → Eligible for promotion
    │        └─ Reject → Status: rejected → Archived, immutable
    │
    ├────→ Promote (move to seed files)
    │        ├─ Symbol validation: warn if symbols unresolved
    │        ├─ Duplicate prevention: check for re-promotion
    │        └─ Status: promoted → Immutable
    │
    ├────→ Regenerate (rebuild from seeds)
    │        ├─ Platform docs: `go run ./tools/api_doc_gen generate platform`
    │        ├─ Site content: `go run ./tools/api_doc_gen generate site`
    │        └─ Seed facts become part of canonical docs
    │
    └────→ GENERATED DOCS (definitive source of truth)
             ├─ API references
             ├─ Caveats & conventions
             ├─ Implementation patterns
             └─ Available to all tools and queries

SEMANTIC CONTRACTS (xml-tree, lua-analysis, xml-lua-links)
    ↓
PARSER/GENERATOR (tools/api_doc_gen)
    ├─ Parser: structural + semantic analysis
    ├─ Classifier: xml-lua validation
    └─ Generator: canonical doc production
    ↓
GENERATED DOCS (source of truth)
```

## Why Observations Are Never Semantic Sources

### Principle 1: Evidence-Backed Input vs. Verified Semantics

**Observations:**
- Backed by addon-level evidence (code examples, UX testing)
- Subject to review by maintainers
- May contain partial/provisional findings
- Confidence explicitly labeled
- Focused on practical addon-development patterns

**Semantic Contracts:**
- Derived from parser analysis of entire codebase
- Generated deterministically from source
- Verified against complete symbol library
- Cover all discovered symbols uniformly
- Single source of truth for API surface

### Principle 2: Governance as Safeguard

Candidate observations **cannot** affect:
- MCP query results
- Doc regeneration (immediate)
- Symbol definitions
- Event flow analysis
- Relationship graphs

Observations only contribute **after** review approval + promotion.

This prevents:
- Untrusted or provisional findings polluting docs
- Unstable symbol definitions
- Inconsistent confidence tracking
- Accidental semantic drift

### Principle 3: Symbols Are Informational in Observations

When observation claims reference symbols via `claims[].symbols[]`:
- These are **informational pointers**, not symbol definitions
- References are validated but **non-blocking**
- Symbols may be forward-references to future docs
- Validation warnings are advisory, not enforcement

Example:
```json
{
  "claim_id": "listdata_binding",
  "statement": "ListData binds ListBox rows to Lua table",
  "symbols": ["ListData", "ListBox"],
  "guidance": "Use table attribute as backing collection path"
}
```

The observation **informs** that ListData and ListBox are related. The parser and generator **determine** the canonical symbol relationships.

## Practical Governance Workflow

### For Addon Developers: Submitting Observations

**When you discover a pattern:**

1. **Document with evidence:**
   ```json
   {
     "entry_id": "myAddon_xml_feature_caveat",
     "status": "candidate",
     "source": {
       "addon": "MyAddon",
       "date_utc": "2026-04-03T10:00:00Z",
       "validation": "dev_observed",
       "context": "window lifecycle issue during form input"
     },
     "claims": [{
       "claim_id": "form_input_lifecycle",
       "kind": "behavior_caveat",
       "confidence": "MEDIUM",
       "statement": "Form input loses focus when nested window re-renders",
       "guidance": "Save form state before window updates; restore after render completes"
     }],
     "evidence": [{
       "evidence_id": "focus_loss_reproduction",
       "type": "code_change",
       "summary": "Workaround in addon maintains input state across re-render",
       "file_refs": ["addons/MyAddon/forms.lua:142"]
     }]
   }
   ```

2. **Create human narrative** for context:
   - What you observed
   - Why it matters
   - How to reproduce
   - Your workaround

3. **Ingest observation:**
   ```powershell
   # Via MCP endpoint
   POST /mcp/ method=feeding/ingest_batch with dry_run=true
   # Review results
   # Then persist with dry_run=false
   ```

4. **Wait for review** — maintainers will accept or reject

**You do NOT:**
- Edit canonical docs directly to include your finding
- Expect immediate visibility in MCP queries
- Assume your evidence will flow into generated docs without review

### For Maintainers: Reviewing Observations

**Decision criteria for accept/reject:**

**ACCEPT if:**
- ✅ Evidence is reproducible (code or test provided)
- ✅ Claim is stated clearly and specifically
- ✅ Pattern is likely to help other addon developers
- ✅ Finding doesn't contradict parser semantics
- ✅ Confidence level matches evidence quality

**REJECT if:**
- ❌ Evidence is sparse or anecdotal only
- ❌ Claim is vague or speculative
- ❌ Finding directly contradicts canonical docs (suggest canonical doc correction instead)
- ❌ Issue is addon-specific, not platform pattern
- ❌ Claim lacks actionable guidance

**Approve workflow:**
```powershell
$payload = @{
  observation_id = "myAddon_xml_feature_caveat"
  verdict = "accept"
  reviewer = "your@email.com"
  notes = "Confirmed against current XE/Coder docs; valuable pattern for form-heavy addons"
}

# Via MCP: feeding/review_observation
```

### For Maintainers: Promoting and Integrating

**After acceptance, decide where this belongs:**

1. **Identify seed file:**
   - Pattern affects XML? → `docs/platform/seeds/xml_conventions.md`
   - Lua-specific? → `docs/platform/seeds/lua_conventions.md`
   - General WAR API fact? → `docs/platform/seeds/war_api_facts.md`

2. **Promote observation:**
   ```powershell
   $payload = @{
     observation_id = "myAddon_xml_feature_caveat"
     dry_run = $false  # After dry-run preview
   }
   # Via MCP: feeding/promote_observation
   ```
   - Symbol validation runs (advisory warnings only)
   - Observation appended to seed file with metadata comment
   - Status changes to `promoted`

3. **Regenerate immediately:**
   ```powershell
   $payload = @{
     scope = "full"
     dry_run = $false
   }
   # Via MCP: feeding/regenerate_from_promoted_knowledge
   ```
   - Platform docs rebuild from seeds
   - Site content regenerates from platform docs
   - Observation's guidance becomes part of canonical docs

4. **Verify in generated docs:**
   ```bash
   grep -r "Form input loses focus" docs/war-api/
   # Should appear in relevant convention pages with source attribution
   ```

## Symbol Validation: Advisory Only

### Why Non-Blocking?

When promoting an observation, the system checks if referenced symbols exist in generated docs.

**Symbols may not resolve because:**
1. Symbol is forward-reference (planned but not yet documented)
2. Symbol exists but under different canonical name
3. Next docs regeneration will add the symbol
4. Symbol is provisional/not part of final API

**Example validation warning:**
```
code: "unresolved_symbol"
message: "symbol 'hypothetical_future_symbol' 
          (referenced in claim) could not be resolved 
          in generated docs. This is advisory; symbols 
          may be forward-references or resolve after docs 
          regeneration. Consider verifying symbol spelling 
          and whether the canonical docs should be updated."
```

**Action:** Warnings are logged but do not block promotion. Maintainer can:
- Verify symbol spelling and update observation if wrong
- Confirm symbol is intentionally forward-referenced
- Update canonical docs if symbol should exist
- Proceed with promotion anyway (symbols are not enforced contracts)

## Regeneration Policy

### When to Regenerate

| Trigger | Scope | Timing | Owner |
|---------|-------|--------|-------|
| Single observation promoted | `full` | Immediately after promotion | Maintainer |
| Batch of observations promoted | `full` | After all promotions complete | Maintainer |
| Seed file manually edited | `full` | Immediately after edit | Maintainer |
| Development/testing | `platform` or `site` | As needed | Developer |
| Scheduled cadence | `full` | Nightly or weekly | CI/automation |

### Regeneration Process

```
feeding/regenerate_from_promoted_knowledge scope=full
    ├─ [Platform Regeneration]
    │   go run ./tools/api_doc_gen generate platform ./docs/addon-api ./docs/war-api
    │   Input: Addon-level API analysis + seed files
    │   Output: docs/war-api/ (canonical platform docs)
    │
    └─ [Site Regeneration]
        go run ./tools/api_doc_gen generate site ./docs/war-api ./docs/site/content
        Input: Platform docs + navigation metadata
        Output: docs/site/content/ (site pages)
```

### Verification After Regeneration

```bash
# 1. Check exit code
if ($regenResult.result.success -eq $true) {
  Write-Host "Regeneration succeeded"
}

# 2. Verify seed content appears in generated docs
grep -r "form-heavy addon" docs/war-api/

# 3. Check file timestamps match regen time
ls -la docs/war-api/*.md | head -5

# 4. Run MCP query to verify updated docs
curl http://localhost:8091/mcp -d '{"method":"search_symbols","params":{"query":"form input"}}'
```

## Seed File Traceability

Promoted observations include HTML metadata for audit trail:

**Format:**
```markdown
<!-- OBSERVATION:entry_id_slug (promoted:2026-04-03T10:30:00Z) -->
> Source: AddonName | Confidence: MEDIUM | Promoted: 2026-04-03

- `MEDIUM`: Claim statement extracted from observation
  - Guidance: Actionable guidance from observation
```

**Benefits:**

1. **Audit Trail:**
   ```bash
   grep -B2 "Form input loses focus" docs/platform/seeds/xml_conventions.md
   # Shows: OBSERVATION:myAddon_xml_feature_caveat (promoted:2026-04-03T10:30:00Z)
   ```

2. **Update Tracking:**
   ```bash
   git log -p docs/platform/seeds/xml_conventions.md
   # Shows which promotions added content
   ```

3. **Deprecation Path:**
   ```bash
   # If pattern becomes outdated, comment references show source
   # Maintainers can mark as deprecated and regenerate
   ```

## Candidate Observations: Isolation by Design

### Why Candidates Don't Affect Docs

1. **Queue Stage Only:**
   - Stored in `accepted.ndjson` review queue
   - Not in seed files
   - Not visible to doc generation

2. **MCP Visibility:**
   - `feeding/list_pending_observations` returns candidates
   - But `search_symbols`, `lookup_symbol`, etc. do NOT see candidates
   - Candidates cannot be queried via standard API

3. **No Regeneration Trigger:**
   - Observation ingestion does NOT trigger regen
   - Manual promotion step required
   - Explicit governance gate

### Example: Candidate Observation Lifecycle

```
Time | Status | Visibility | Doc Impact
-----|--------|------------|----------
T0   | candidate | Queue only | None
     | Reviews pending...
T1   | reviewed (accepted) | Queue only | None
     | Awaiting promotion decision...
T2   | promoted | Seed file | Docs updated on next regen
T3   | (after regen) | Docs + MCP | Full visibility
```

Until **T2**, docs are unaffected. This prevents:
- Untrusted findings polluting canonical output
- Accidental symbol definition conflicts
- Inconsistent confidence tracking
- Breaking changes to API surface

## Best Practices

### For Addon Developers

1. **Be specific with evidence:**
   - Include file paths and line numbers
   - Link to reproducer code in your addon
   - Document what you tried and failed

2. **Label confidence honestly:**
   - `HIGH`: Works reliably, cross-validated
   - `MEDIUM`: Observed in own addon, reasonable theory
   - `LOW`: Theory only; needs more testing

3. **Provide actionable guidance:**
   - Not just "this is broken"
   - But "here's the workaround and why it works"
   - Explain when addon developers need this pattern

4. **Use standard claim kinds:**
   - `behavior_caveat`: XML/Lua quirk you found
   - `usage_pattern`: Idiom that works well
   - `signature_hint`: Symbol exists but undocumented

### For Maintainers

1. **Cross-validate before promotion:**
   - Reproduce finding in your own test addon
   - Check if canonical docs should be updated instead
   - Verify evidence is repeatable

2. **Link observations to symbol updates:**
   - If observation reveals missing symbol, create symbol doc first
   - THEN promote observation (symbols will resolve)
   - Regenerate after both steps complete

3. **Monitor rejection reasons:**
   - Trends in rejected observations guide doc improvements
   - Frequent rejections of certain kinds → docs unclear?
   - Use rejection feedback to improve canonical docs

4. **Batch promotions for efficiency:**
   - Collect 5-10 accepted observations
   - Promote as batch to same seed file
   - Regenerate once (faster than individual regen)

### For CI/Automation

1. **Schedule regular regenerations:**
   ```bash
   # Daily at 2 AM UTC
   0 2 * * * cd /workspace && make generate-platform && make generate-site
   ```

2. **Monitor regeneration failures:**
   - Alert if `regenerate_from_promoted_knowledge` errors
   - Check for seed file syntax errors
   - Verify parser/generator executables available

3. **Backup before promotion:**
   ```powershell
   # Automated backup of docs/ before batch promotions
   tar czf backup-$(date +%s).tar.gz docs/
   ```

## Troubleshooting

### Symbol Validation Warnings Don't Stop Promotion

**Question:** My observation has unresolved symbol warnings. Should I fix it?

**Answer:** The warning is advisory. You can:
1. Update observation to remove or correct symbol name
2. Create missing symbol doc in canonical docs, then promote
3. Promote anyway (symbol may be forward-reference)

Choose based on whether symbol should exist in API.

### Candidate Observation Not Showing in Doc Queries

**Question:** I ingested an observation but `search_symbols` doesn't find it.

**Answer:** Correct behavior. Observations affect docs only after:
1. Status: `reviewed` (via `review_observation` with verdict=accept)
2. Status: `promoted` (via `promote_observation`)
3. Docs regenerated (via `regenerate_from_promoted_knowledge`)

Use `feeding/list_pending_observations` to see candidates.

### Promoted Observation Doesn't Appear in Regenerated Docs

**Question:** I promoted an observation and regenerated, but don't see it in docs.

**Diagnosis:**
1. Check promotion succeeded: `feeding/list_pending_observations` shows status=`promoted`
2. Verify seed file was updated: `grep OBSERVATION: docs/platform/seeds/`
3. Confirm regeneration succeeded: `regenerate_from_promoted_knowledge` reported `success=true`
4. Check file timestamps: docs should be newer than promotion time

**Action:**
```bash
# Manual regeneration with verbose output
make generate-platform 2>&1 | tail -20
make generate-site 2>&1 | tail -20
```

### Duplicate Promotion Error

**Question:** I get "entry is already promoted" when promoting again.

**Answer:** This is correct. Promoted observations are immutable. If you need to update:
1. Create new observation with updated information
2. Use different entry_id (e.g., `v2` suffix)
3. Promote new observation
4. Optionally mark old observation as superseded in seed file

## Technical Details

### Observation Schema Fields

For reference in observation creation:

```json
{
  "schema_version": "feeding.observation.v1",
  "entry_id": "slug",
  "status": "candidate",
  "source": {
    "addon": "AddonName",
    "date_utc": "RFC3339 timestamp",
    "validation": "dev_observed|user_validated|mixed",
    "context": "Brief context of discovery"
  },
  "target_seeds": [
    "docs/platform/seeds/xml_conventions.md"
  ],
  "confidence": {
    "overall": "HIGH|MEDIUM|LOW",
    "rationale": "Why this confidence level"
  },
  "claims": [
    {
      "claim_id": "unique_id",
      "kind": "behavior_caveat|usage_pattern|signature_hint",
      "confidence": "HIGH|MEDIUM|LOW",
      "statement": "Clear claim statement",
      "guidance": "How addon developers should apply this",
      "symbols": ["SymbolName"],
      "evidence_refs": ["evidence_id_1"]
    }
  ],
  "evidence": [
    {
      "evidence_id": "unique_id",
      "type": "code_change|runtime_observation|user_report|test",
      "summary": "What this evidence shows",
      "file_refs": ["path/to/file:line"]
    }
  ],
  "promotion": {
    "notes": "When to promote, any caveats",
    "criteria": ["List of promotion acceptance criteria"]
  }
}
```

### Queue File Format

NDJSON (newline-delimited JSON):

```json
{"ingested_at_utc":"2026-04-03T10:00:00Z","source_path":"docs/platform/feeding/xml/myPattern.feed.json","observation":{...},"lifecycle_status":"candidate"}
{"ingested_at_utc":"2026-04-03T11:00:00Z","source_path":"docs/platform/feeding/xml/myPattern.feed.json","observation":{...},"lifecycle_status":"reviewed","review":{"verdict":"accept","reviewer":"user@example.com","notes":"Validated","reviewed_at_utc":"2026-04-03T11:15:00Z"}}
```

Each line is a complete observation lifecycle record.

## Summary

The strengthened feeding mechanism provides **governance-backed addon discovery intake** while maintaining semantic contracts as the sole source of truth.

**Key Points:**
- ✅ Observations are reviewed input, not semantic sources
- ✅ Candidates don't affect docs (immutable until reviewed+promoted)
- ✅ Symbol validation is advisory (non-blocking)
- ✅ Seed traceability enables audit trail
- ✅ Regeneration transforms promoted observations into canonical docs
- ✅ Clear governance gates prevent unstable findings polluting semantics

**Workflow:** Ingest → Review → Approve → Promote → Regenerate → Published

**Safety:** Each step requires explicit action; no automatic propagation until final promotion.
